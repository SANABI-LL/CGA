CampusGeo · Backend Fix Spec Spec 03 · Compound Queries

Compound
Filter + Proximity

“Green ash trees near Regenstein” needs a species filter and a spatial buffer in one query. Today the agent picks one tool and drops the other. This spec adds compound-query support to handler.js.

ENVIRONMENT  Claude Code
FILE        campusgeo-lambda/handler.js
DEPLOY      update-function-code
SCOPE       1 tool + 1 executor
01

The bug

× Today
“show me all green ash trees near Regenstein library”

Bedrock sees two candidate tools — filter_features (species) and features_within (proximity) — and can only call one.

It picks filter_features → returns every green ash on campus. The “near Regenstein” half is silently dropped.

✓ Expected
“show me all green ash trees near Regenstein library”

One tool runs both constraints: filter trees where CommonName ⊃ "Ash-Green", then keep only those inside a buffer around Regenstein.

Result: the handful of green ash within ~150m of the Reg — not the whole campus.

This is the general class of compound spatial query: attribute filter × spatial constraint. It recurs constantly (“owned buildings near the Midway”, “poor-condition trees within 100m of Ellis”), so fix it structurally, not case by case.

02

Approach: extend features_within

Two ways to solve this: (a) let the agent chain tools across turns, or (b) give features_within an optional filters parameter so one call does both. Choose (b) — it is one tool call, one round-trip, cheaper and more reliable than multi-turn tool chaining. The filter logic already exists in the filter_features executor; we reuse it.

03

Change 1 · Tool schema

campusgeo-lambda/handler.js

Add an optional filters array to the features_within tool spec. Same filter shape as filter_features, so Bedrock already knows how to fill it.

{ toolSpec: { name: 'features_within',
  description: 'Find features within a distance of an anchor location. Optionally filter

  feature description ("green ash trees", "owned buildings").',

04

Change 2 · Executor

campusgeo-lambda/handler.js · executeIntent()

In the case 'features_within' branch, after building the buffer and collecting features inside it, apply the same filter predicate used by filter_features. Extract that predicate into a shared helper so both tools call it.

// ── Shared filter predicate (hoist to module scope; used by both tools) ──
function matchesFilters(props, filters, match) {
  if (!filters || !filters.length) return true;
  const results = filters.map((flt) => {
    const v = props[flt.field];
    switch (flt.op) {
      case 'eq':     return String(v).toLowerCase() === String(flt.value).toLowerCase();
      case 'neq':    return String(v).toLowerCase() !== String(flt.value).toLowerCase();
      case 'gt':     return +v > +flt.value;
      case 'lt':     return +v < +flt.value;
      case 'contains': return String(v||'').toLowerCase().includes(String(flt.value).toLowerCase());
      case 'between':  return +v >= flt.value[0] && +v <= flt.value[1];
      default:    return true;
    }
  });
  return match === 'any' ? results.some(Boolean) : results.every(Boolean);
}

// ── In case 'features_within': ──────────────────────────────────────────
case 'features_within': {
  const anchor = findByName(buildings.features, input.anchor_name);
  if (!anchor) return { answer: `Could not find "${input.anchor_name}".`, features: null, mapAction: null };
  const pt  = turf.centroid(anchor);
  const radius = input.distance_m || 150;  // sensible default
  const buf = turf.buffer(pt, radius / 1000, { units: 'kilometers' });
  const tgt = input.target_layer === 'trees' ? trees : buildings;

  const hits = tgt.features
    .filter((f) => turf.booleanPointInPolygon(turf.centroid(f), buf))  // spatial
    .filter((f) => matchesFilters(f.properties, input.filters, input.match)); // attribute

  const desc = (input.filters && input.filters.length)
    ? input.filters.map((f) => `${f.field} ${f.op} ${f.value}`).join(', ')
    : (input.target_layer || 'features');

  return {
    answer: `${hits.length} ${input.target_layer||'features'} within ${radius}m of ${nameOf(anchor)}${(input.filters&&input.filters.length)?` matching ${desc}`:''}.`,
    features: { type: 'FeatureCollection', features: hits },
    mapAction: { type: 'buffer', center: pt.geometry.coordinates, radiusM: radius }
  };
}

Refactor note: the existing case 'filter_features' branch has this same switch inline — replace it with a call to matchesFilters(f.properties, input.filters, input.match) so there is one source of truth.

Change 3 · System prompt

campusgeo-lambda/handler.js · SYSTEM

Add one line so Bedrock reliably reaches for the compound tool. Append to the existing system prompt:

// append to SYSTEM string:
`
When a query combines a LOCATION ("near X", "within N metres of X", "around X")
with a feature description ("green ash trees", "owned buildings", "trees in poor
condition"), you MUST call features_within with BOTH distance and the filters array.
Do not call filter_features for these — it ignores location and returns the whole campus.
Species live in CommonName as "Type-Variety" (e.g. green ash = "Ash-Green",
use op "contains" with value "Ash-Green"). Default distance_m to 150 if unstated.`
06

Species naming reference

The CommonName field stores species as Type-Variety. Users say them the other way round. Bedrock handles most of this, but if matching is unreliable, embed this mapping in the system prompt or a lookup. Confirm exact strings against your real trees.geojson.

// user phrase → CommonName contains
"green ash"         → "Ash-Green"
"norway maple"     → "Maple-Norway"
"littleleaf linden" → "Linden-Littleleaf"
"honey locust"     → "Locust-Honey"
"red oak"         → "Oak-" (Red / Northern)
