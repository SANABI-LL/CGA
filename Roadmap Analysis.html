Daily Data-Update Pipeline

数据自己照顾自己。

每天凌晨自动检测、校验、更新 —— 带 LLM 质检步骤的自愈数据管道。本身就是一个独立的 portfolio 故事。

1

EventBridge 定时触发

daily cron · 02:00 CT —— 检查 S3 上传桶中的新地理数据库导出

2

Diff 检测

对比要素数量、属性哈希、几何校验和 —— 无变化则静默退出

3

校验 + Claude 质检

schema / CRS / 坐标合理性校验；Claude 总结变更集并标记异常 —— 坐标漂移、字段缺失、可疑删除

正常

发布更新

写入 S3 → 更新 DynamoDB 时间戳 → CloudFront 缓存失效 → SNS 通知："3 栋建筑更新，12 棵树新增"

异常

挂起待审

暂停更新，通知人工复核 —— 数据永远不会带病上线

User Memory

历史是记录，画像是记忆。

两个独立的特性：query_history 表记录每一次查询；定时 Lambda 用 Claude 把它蒸馏成紧凑的用户画像。

查询历史

每次查询 + GeoJSON 结果写入 query_history，支持一键重放到地图

Claude 蒸馏

定时 Lambda 定期总结查询模式，生成紧凑画像存入 user_profiles

"频繁查询北校区建筑状况；春季偏好树木图层"

个性化输出

画像驱动个性化查询建议与数据变更通知 —— "您关注的 3 栋建筑 RI 值发生了变化"

预算 ~10 小时 的轻量特性 —— 对 AI 岗位 portfolio 叙事的性价比极高。

Known Constraints

已知的边界，写在前面。

Lambda 冷启动

冷启动需从 S3 拉取并解析 GeoJSON，增加 1–2s 延迟。对策：module-scope 加载 + /tmp 缓存，演示前预热；<2s p95 目标仅适用于 warm 状态。

大数据集查询性能

>10MB GeoJSON 无内建空间索引。对策：Phase 3 实现 R-tree 索引；当前 1.7MB 数据量在最舒适区间。

ETL 维护成本

地理数据库更新需要持续维护转换管道。对策：Phase 4 的每日自动更新 pipeline 将人工干预降至异常复核。

LLM 空间推理边界

"理想位置"类查询产出的是评分候选清单，不是可证明的最优解。对策：明确定位为决策支持工具；真正的优化模型（Location-Allocation / MILP）留给 Phase 4+。

数据缺口

FAR 数据可能需手动补充属性；实时数据（如 Divvy 站点）需外部 API 集成。对策：列入 Phase 3 数据准备工作。
